NYU · Protocols.io Project

Protocols Assistant — Architecture, Design, & LLM Usage

A conversational research assistant powered by Claude/Ollama LLM for intent extraction and explanation, combined with rule-based ranking for deterministic, transparent protocol recommendations.

Branch: v4 Deployed on Google Cloud Run June 2026 ▶ Live demo

📋 1 · Executive Summary

A system that lets scientists describe an experiment in everyday language and receive the most relevant step-by-step protocols, blended with supporting literature. Guided conversation extracts a structured experiment profile (organism, technique, delivery method, readout, …), driving search across 22,670 protocols.io protocols and live PubMed. Results merged using 100% rule-based ranking (deterministic, transparent, cost-effective). LLM used strategically for 3 tasks only: intent extraction, candidate query generation, and result explanation (optional). Deployed as single container on Google Cloud Run.

🤖 5 · LLM Usage Strategy

LLM is called sequentially through the conversation until search is confirmed. Ranking stays 100% rule-based for determinism and cost.

LLM Call Sequence (Timeline)

Stage Function When Latency
A analyze_experiment_request() Initial query + each clarification answer (until profile complete) ~5s per call
B generate_natural_search_queries() After all clarifications done + profile complete ~5s
C explain_matches() After search results returned (if user requests explanation) ~0.5s

Detailed Flow

1. Initial query: Call A runs once → extracts intent, builds profile, checks for clarifications needed

2. Clarification loop: Call A runs again for EACH user answer → re-analyzes with updated context → checks if more clarifications needed

3. When profile complete: Call B runs once → generates 5 candidate search queries

4. User selects query: search_confirmed=true → LLM stops running → search executes (no LLM)

5. After results: Call C runs if user requests explanation (optional)

Total LLM Cost

  • 0 clarifications: 2 calls (A + B) = ~10 seconds before search starts
  • 2 clarifications: 4 calls (A×3 + B) = ~20 seconds before search starts
  • Then search runs (no LLM, parallel 15 variants)

What Does NOT Use LLM

  • Query expansion: Rule-based (3 variants per query)
  • Result ranking: 100% rule-based weighted signals
  • Protocols.io/PubMed search: Direct API calls
  • Local TF-IDF search: Statistical algorithm
  • Result blending: Title-weighted merge
Design: LLM-first for profile extraction (NLU strength). Rule-based for ranking (determinism, transparency, free). Search runs after profile is complete.

⚖️ 6 · Ranking Logic (Rule-Based)

100% deterministic two-tier scoring using weighted signals. Same query always produces same ranking.

Tier 1: Query-Based (TF-IDF + Signals)

score = (title×0.25 + organism×0.20 + method×0.15 + desc×0.15 + synonym×0.10 + freq_bonus×0.05 + coverage×0.10) × penalty

Penalty = 0.1 if required molecule missing, else 1.0.

Tier 2: Profile-Based Re-Ranking (if profile exists)

profile_score = base + (organism×2.4 + method×1.8 + expression×2.2 + combined×3.2 + required_concept×2.8 + ...) × penalty
Why rule-based? Determinism (reproducible testing), transparency (explain exact score), speed (instant), cost (free).

🔍 7 · Search & Ranking Pipeline

Query Variants (Rule-Based)

Each query expanded into 3 variants: Initial (original), Level 1 (full terms), Level 2 (stripped).

Select 1 query = 3 variants. Select all 5 = 15 variants fired in parallel via ThreadPoolExecutor(max_workers=8).

Ranking Steps

  1. Tier 1: Score by TF-IDF, organism, method, concept coverage, frequency.
  2. Tier 2: Re-score by profile field matches (if profile available).
  3. Filter: Remove severely off-target (penalty ≤ 0.4).
  4. Blend: Merge with PubMed using title-weighted ranking.
  5. Explain (opt): LLM Call #3 generates explanation.

📌 9 · LLM vs Rule-Based Quick Reference

Generate candidate queries generate_natural_search_queries() ✓ Domain-aware generation

Explain results explain_matches() ✓ Natural language

⚡ 10 · Performance & Cost

Latency Breakdown (Search All)

Total ~2-3s = LLM intent (1-2s) + LLM queries (1-2s) + parallel_search (2-3s) + blend (<0.1s)

Cost-Benefit

  • LLM calls: 2-3 per search (Call #3 optional). Minimal API overhead.
  • Rule-based ranking: Instant, free, repeatable (no model drift).
  • Parallelization: 15 protocols.io queries in ~2-3s vs ~20-25s sequential.
  • No hallucination: Ranking deterministic; LLM only for intent & explanation.
Hybrid approach: Use LLM where it excels (NLU, NLG), keep ranking rule-based for reliability and cost.
