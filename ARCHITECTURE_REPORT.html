NYU · Protocols.io Project

Protocols Assistant — Architecture & Design

A conversational research assistant that turns a plain-language experiment description into a ranked, blended set of laboratory protocols and peer-reviewed literature.

Branch: v4 Deployed on Google Cloud Run June 2026 ▶ Live demo

📋 1 · Executive Summary

I built a system that lets a scientist describe an experiment in everyday language and receive the most relevant step-by-step protocols, blended with supporting literature. A short guided conversation extracts a structured experiment profile (organism, technique, delivery method, readout, …), which then drives a search across two complementary sources: a local corpus of 22,670 protocols.io protocols and a live PubMed query. Results from both are merged into one title-weighted, source-tagged ranked list. The entire application — backend, conversational layer, and UI — runs as a single container on Google Cloud Run.

🏗️ 3 · High-Level Architecture

The system is deployed as a single container on Cloud Run that serves both the FastAPI backend and the static chat UI from the same origin — one URL, no CORS or cross-service wiring. The protocol corpus and a prebuilt search index are baked into the image at build time, so the service is ready the instant it starts.

🔄 4 · Conversational Flow

Rather than asking the scientist to write a perfect query, I extract intent through a short guided conversation, then build the query from the structured profile. The deterministic search runs only once the profile is ready.

1 Request Plain-language experiment description.
→
2 Profile LLM planner extracts a structured experiment profile.
→
3 Clarify One field at a time; handles declines & AND/OR/LIKE.
→
4 Queries Candidate searches, most-specific-first.
→
5 Blend protocols.io + PubMed merged & ranked.
→
6 Explain Rationale, match notes, and model used.

🧩 6 · On MCP (Model Context Protocol)

I evaluated using a ready-made PubMed MCP server versus implementing PubMed access natively, and chose native Python. The MCP server is a local Node process, whereas my pipeline is deterministic (no agentic tool-loop) and the deployment is a single Python container — so MCP would have added a runtime and indirection for no relevance gain. I kept its full-text fallback chain (PMC → Europe PMC → Unpaywall) as a design reference rather than a dependency.

🎯 7 · Ranking & Blending

Two very different engines feed one ranked list. Each source is retrieved according to its own nature, but both are scored on a single title-weighted axis so the blend is fair.

PubMed (literature) protocols.io (protocols)
# one comparable score across both sources
score = 0.67 · cosine(query, title) + 0.33 · cosine(query, body)
  • protocols.io: TF-IDF relevance (with a title boost) plus closeness tiers — organism (species > family > kingdom), method, and tissue match.
  • PubMed: a trimmed core-concept query (so its boolean-AND search isn't over-constrained), with OR translated to a true disjunction.
  • Blend: protocols.io keeps its own order; PubMed papers are interleaved by the unified score, and every result is PubMed / protocols.io tagged. Default 5 per source, configurable.
Key choice: ranking and blending are fully deterministic — the LLM shapes the conversation and the query, but never ranks results, so output is reproducible and does not swing with model choice.
